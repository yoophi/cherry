# Cherry 클론 코딩 구현 가이드

## 목표 해석

이 프로젝트는 "서울 벚꽃 지도"를 보여주는 정적 웹 서비스다.  
핵심은 예쁜 소개 페이지가 아니라, **지도 위에 많은 포인트 데이터를 보기 좋게 올리고 필터링하는 경험**이다.

## 우선순위

클론 코딩 우선순위는 아래 순서가 맞다.

1. 전체 레이아웃 재현
2. Leaflet 지도 초기화
3. 나무 마커 표시
4. 마커 클러스터링
5. 구별 필터
6. 히트맵
7. 팝업/애니메이션/개화 상태 같은 디테일

## 화면 구성

### 상단 헤더

- 벚꽃 아이콘
- 제목
- 서브타이틀

### 좌측 사이드바

- 개화 상태 배너
- 전체 통계 배너
- 구 필터 버튼 그룹
- 구별 리스트
- 데이터 출처 노트

### 메인 영역

- Leaflet 지도
- 우측 상단 안내 박스
- 하단 외부 링크 배너

## 기능 단위 분해

### 1. 지도 초기화

- Leaflet map 생성
- 서울 중심 좌표로 초기화
- OpenStreetMap 타일 적용

### 2. 데이터 모델

원본에서 보이는 데이터 형태:

- `RAW`: 개별 나무 좌표 목록
- `STATS`: 일부 구의 확보 좌표 수
- `CENSUS`: 구별 전체 통계
- `SPOTS`: 주요 명소 수동 데이터

권장 타입 예시:

```js
{
  lat: 37.55,
  lng: 126.98,
  district: "영등포구",
  road: "윤중로",
  height: 5,
  diameter: 23
}
```

### 3. 마커 계층

원본은 지도 위에 3종류 마커를 올린다.

- 개별 나무 마커
- 명소 마커
- 통계 보정 마커

이 3개를 분리해서 관리해야 필터링이 쉽다.

### 4. 필터링 로직

구 필터 선택 시 해야 할 일:

- 클러스터 그룹 재구성
- 히트맵 데이터 재생성
- 명소/통계 마커 표시 여부 갱신
- 리스트 active 상태 갱신
- 필요 시 지도 bounds 이동

## 원본 구현의 장단점

### 장점

- 단일 파일이라 빠르게 배포 가능
- 외부 의존성이 적음
- 데이터가 작동 시점에 바로 준비되어 초기 구현이 쉬움

### 단점

- HTML 파일이 매우 커짐
- 데이터 수정이 어려움
- 기능 확장 시 유지보수성이 급격히 나빠짐
- 테스트/재사용 구조가 약함

## 클론 코딩 시 권장 개선점

원본을 그대로 베끼기보다 아래는 개선하는 편이 좋다.

- 데이터 JSON 분리
- JS 모듈 분리
- 스타일 파일 분리
- 상수와 UI 렌더 함수를 분리
- 마커 생성기와 필터 로직을 별도 함수로 분리

예시 구조:

```text
docs/
public/
  index.html
src/
  app.js
  map.js
  filters.js
  data.js
  ui.js
data/
  trees.json
  census.json
  spots.json
styles/
  main.css
```

## 기술 선택 권장안

### 원본에 가깝게 만들려면

- `Vanilla JavaScript + Leaflet`

### 유지보수성까지 챙기려면

- `Vite + Vanilla JS` 또는 `Vite + React`

다만 원본 자체는 프레임워크 앱이 아니라서, **정확한 clone coding**이 목적이면 Vanilla JS가 더 가깝다.

## 결론

이 프로젝트의 본질은 다음이다.

- 정적 페이지
- 지도 중심 서비스
- 좌표 데이터 시각화
- Leaflet 기반 인터랙션
- 프레임워크 없는 단일 파일 앱

따라서 첫 구현은 과하게 추상화하지 말고, **정적 데이터 + Leaflet + 간단한 상태 변수** 중심으로 만드는 것이 가장 빠르고 원본에도 가깝다.
